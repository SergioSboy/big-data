## 1. Цель работы

Построить end-to-end пайплайн **Bronze → Silver → Gold** на реальном датасете задержек рейсов США (аналог данных [Kaggle: Flight Delay Analysis 2018–2024](https://www.kaggle.com/code/peymanradmanesh/flight-delay-analysis-2018-2024)), с последующим обучением моделей **регрессии** (прогноз `ARR_DELAY`, минуты) и **классификации** (бинарный признак `is_delayed`), логированием экспериментов в **MLflow** и использованием возможностей **Delta Lake** (включая MERGE, оптимизации и time travel).

**Хранилище:** локальные каталоги на диске (`data/delta`, монтируются в контейнеры), без MinIO — допустимый вариант по условию задачи.

---

## 2. Исходные данные и постановка ML

- Источник: CSV в стиле US DOT / BTS (поля вроде `Year`, `Month`, `Origin`, `Dest`, `ArrDelay`, `Cancelled` и т.д.), файл `data/flight_data_2018_2024.csv`.
- **Регрессия:** целевая переменная — `ARR_DELAY` (задержка прибытия, минуты).
- **Классификация:** `is_delayed = 1`, если `ARR_DELAY > 15` минут, иначе `0` (порог задан при построении Gold feature table).
- **Разделение выборок:** обучение на рейсах с `YEAR < 2024`, тест на `YEAR == 2024` (при отсутствии данных за 2024 возможен fallback на случайный сплит в коде).

Фактические объёмы при успешном прогоне ML (лог `run_ml.py`):

| Выборка | Число строк |
|--------|-------------|
| Train  | 1 783 943   |
| Test   | 446 097     |

## 4. Слой Bronze

**Назначение:** загрузка сырых данных в Delta-таблицу с имитацией инкрементального поступления.

- Чтение CSV, приведение к **канонической схеме** (в т.ч. вариант с колонкой `FlightDate` из датасета Kaggle).
- Загрузка **по годам** отдельными батчами, режим **`append`**, опция **`mergeSchema=true`** — версионирование таблицы и **schema evolution**.
- После каждого годового батча демонстрируется **time travel:** чтение снимка таблицы **`versionAsOf`** предыдущей версии (до текущего append).
- После полной загрузки диапазона лет: **OPTIMIZE (compaction)** и **Z-ORDER** по `YEAR`, `ORIGIN_AIRPORT`.

Если CSV не найден, код может сгенерировать один синтетический батч (демо-режим) без годового разбиения.

---

## 5. Слой Silver

**Назначение:** очищенная и обогащённая таблица для аналитики и Gold.

- **Очистка:** исключение отменённых рейсов (`CANCELLED == 0`), ограничение выбросов по задержкам (например, отсечение экстремальных значений), нормализация строковых полей (например, `AIRLINE` в верхний регистр), работа с пропусками.
- **Производные признаки:** `hour` из планового времени вылета, `day_of_week`, `season`, маршрут `route` (конкатенация аэропортов), флаги `is_weekend`, `is_rush_hour`.
- **Запись:** партиционирование **`partitionBy=["YEAR", "MONTH"]`**.
- **Идемпотентность повторных прогонов:** обновление через **`MERGE`** по ключу рейса (дата, плановое время, аэропорты, авиакомпания, номер рейса), без неограниченного дублирования строк.
- После записи: **OPTIMIZE**, **Z-ORDER** по `AIRLINE`, `ORIGIN_AIRPORT`, **VACUUM** (retention 168 ч; в Spark отключена строгая проверка минимального срока хранения для учебного сценария).

## 6. Слой Gold

Две группы объектов:

1. **Аналитические витрины** (агрегаты по аэропорту, авиакомпании, часу, сезону и т.д.) — режим **`overwrite`**, при повторном запуске таблицы пересоздаются без накопления дублей логики витрин.
2. **Feature table для ML:** числовые и категориальные поля, **StringIndexer + OneHotEncoder** для категорий, **VectorAssembler** → колонка `features`; целевые `ARR_DELAY` и `is_delayed`; запись в Delta с партицией по **`YEAR`**.

## 7. Polars lazy и пушдауны

После записи Silver выполняется демонстрационный запрос: **`pl.scan_delta(silver_path)`** → фильтры по году и задержке → **`group_by`** → агрегаты → **`sort`**, вывод плана через **`LazyFrame.explain()`**.

Типичный фрагмент плана (имена узлов могут немного отличаться в зависимости от версии Polars):

```text
1. SELECT [col("ARR_DELAY"), col("AIRLINE"), col("season"), col("YEAR")]
2. FILTER [(col("ARR_DELAY")) > (0)]
   FROM
     FILTER [(col("YEAR")) == (2024)]
     FROM
       SCAN [projected 4 columns]
```

**Обоснование партиций Silver `YEAR`, `MONTH`:** запросы с фильтром по времени отсекают каталоги партиций; в сочетании с **Z-ORDER** по перевозчику и аэропорту вылета снижается объём сканируемых данных при аналитике и построении Gold.

## 8. Возможности Delta Lake (выполнение требований)

| Возможность | Где используется |
|-------------|------------------|
| **MERGE** | Silver — обновление без дублирования ключей рейса |
| **OPTIMIZE (compaction)** | Bronze (в т.ч. после батчей), Silver |
| **Z-ORDER** | Bronze (`YEAR`, `ORIGIN_AIRPORT`); Silver (`AIRLINE`, `ORIGIN_AIRPORT`) |
| **VACUUM** | Silver |
| **Time travel** | Bronze — `versionAsOf` после годовых append |
| **Schema evolution** | Bronze — `mergeSchema`; Gold — `overwriteSchema` при перезаписи витрин |

## 9. Эксперименты ML и результаты

### 9.1. Модели

- **Регрессия:** LinearRegression, RandomForestRegressor, GBTRegressor.
- **Классификация:** LogisticRegression (с подбором гиперпараметров через **CrossValidator**), RandomForestClassifier.

Метрики: для регрессии — RMSE, R², MAE; для классификации — accuracy, F1, AUC (где применимо).

### 9.2. Сводная таблица метрик (тестовая выборка, фактический лог)

**Регрессия (`ARR_DELAY`):**

| Модель | RMSE | R² | MAE |
|--------|------|-----|-----|
| LinearRegression | 15.78 | 0.874 | 10.76 |
| RandomForest | 22.78 | 0.736 | 14.94 |
| GBTRegressor | 18.49 | 0.826 | 11.92 |

**Классификация (`is_delayed`):**

| Модель | Accuracy | F1 | AUC |
|--------|----------|-----|-----|
| LogisticRegression | 0.911 | 0.905 | 0.922 |
| RandomForestClassifier | 0.771 | 0.671 | 0.861 |

### 9.3. Краткая интерпретация

- По **регрессии** наиболее точной оказалась **линейная модель** (минимальный RMSE, максимальный R²). Ансамбли на one-hot признаках показали более высокую ошибку — типичная ситуация при сильной разреженности и линейно-аддитивной структуре таргета после кодирования.
- По **классификации** **логистическая регрессия** заметно превосходит случайный лес по accuracy, F1 и AUC на используемом наборе фич.
- **Важность признаков** для Random Forest выводится по **индексам компонент вектора `features`** (`idx_0`, `idx_5`, …) — это позиции после сборки пайплайна (числовые признаки и закодированные категории), а не исходные имена колонок в сыром CSV.

## 10. MLflow

- Сервис **MLflow** запускается отдельным контейнером; UI на хосте: **`http://localhost:5050`** (порт задаётся переменной `MLFLOW_HOST_PORT`, по умолчанию 5050).
- В каждом run логируются **параметры** (тип задачи, алгоритм, размеры выборок и др.), **метрики**, **артефакты Spark ML-моделей**, **версия Delta-таблицы Gold** (`gold_table_version` через историю таблицы).
- Tracking URI внутри Docker: `http://mlflow:5000`. Артефакты хранятся в общем томе `./mlflow_data` (согласованность путей с сервером MLflow).
